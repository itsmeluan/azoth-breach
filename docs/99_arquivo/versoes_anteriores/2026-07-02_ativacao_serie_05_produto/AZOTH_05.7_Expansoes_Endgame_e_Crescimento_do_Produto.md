# AZOTH: BREACH — 05.7 — Expansões, Endgame e Crescimento do Produto

- Camada: `05_produto`
- Tipo: Estratégia de crescimento, expansão e continuidade de produto
- Status: Documento de referência operacional
- Dependências principais: `00`, `01`, `02`, `03`, `04`, `05.0` a `05.6`
- Não substitui: documentação canônica, sistemas de gameplay, design detalhado de missões, tuning numérico, roteiro narrativo ou plano técnico de implementação

---

# 1. Função deste Documento

Este documento define como AZOTH: BREACH deve crescer depois do primeiro recorte jogável, do MVP, da vertical slice e do lançamento inicial.

Seu objetivo é estabelecer diretrizes para:

- expansões de regiões;
- expansão de campanha;
- expansão de endgame;
- adição de novas Academias, personagens e operações;
- crescimento de sistemas sem inflação descontrolada;
- evolução de temporadas para conteúdo permanente;
- novos modos de jogo;
- integração entre live ops, monetização e identidade canônica;
- critérios para decidir quando expandir, quando cortar e quando não crescer.

Este documento não cria conteúdo final de expansão. Ele define **como o produto pode crescer sem perder a própria forma**.

---

# 2. Princípio Central

AZOTH: BREACH só deve crescer quando a expansão aprofundar a identidade do jogo.

Crescimento aceitável:

- revela novas camadas de Aletheia;
- amplia leitura de Breaches;
- cria novas decisões alquímicas;
- aumenta variedade tática;
- permite builds mais autorais;
- abre novas frentes de investigação;
- fortalece o Codex Veritatis;
- renova grind e endgame com significado;
- respeita custo, risco, incompletude e consequência.

Crescimento ruim:

- adiciona volume sem função;
- cria sistemas paralelos que ignoram transmutação;
- transforma Breach em desculpa para qualquer evento;
- banaliza revelações;
- infla poder vertical sem controle;
- exige produção maior do que o projeto consegue sustentar;
- aumenta complexidade sem melhorar decisão;
- adiciona conteúdo apenas para manter calendário.

A regra é simples:

> **AZOTH deve expandir por profundidade antes de expandir por quantidade.**

---

# 3. Relação com a Estratégia de Produto

A camada `05` já define que AZOTH: BREACH é um produto mobile-first, tático, narrativo, modular e produzido com forte atenção à viabilidade.

Este documento parte das seguintes decisões fechadas:

- o jogo não deve começar como mundo aberto;
- o jogo não deve depender de multiplayer para funcionar;
- o jogo não deve monetizar poder estrutural;
- temporadas não substituem campanha;
- live ops só existe se o jogo base já for forte;
- grind precisa ser sempre útil, mas não pode destruir a campanha;
- Codex é sistema de progressão, não biblioteca passiva;
- expansão deve preservar cânone, risco e incompletude.

Logo, expansão em AZOTH não significa “mais tudo”.

Significa:

> **mais formas de operar, investigar, interpretar, construir e sobreviver dentro das regras do universo.**

---

# 4. Tipos de Crescimento do Produto

O crescimento de AZOTH deve ser dividido em categorias claras.

## 4.1 Crescimento Horizontal

Adiciona novas opções sem aumentar drasticamente o poder bruto.

Exemplos:

- novas ETs;
- novos modificadores;
- novas runas;
- novos catalisadores;
- novos objetivos de operação;
- novas regiões;
- novos tipos de Breach;
- novas condições de campo;
- novas entradas de Codex;
- novos personagens jogáveis ou de suporte.

Este é o crescimento mais saudável para o jogo.

## 4.2 Crescimento Vertical Controlado

Aumenta poder de forma limitada e mensurável.

Exemplos:

- aumento pequeno de proficiência;
- upgrade de instrumentos;
- expansão de complexidade alquímica;
- melhoria de eficiência em tipos específicos de operação;
- acesso a conteúdo mais difícil.

Este crescimento deve existir, mas com teto.

## 4.3 Crescimento Narrativo

Amplia compreensão do universo.

Exemplos:

- novas campanhas;
- novas linhas de investigação;
- revelações parciais;
- arquivos acadêmicos;
- ruínas pré-modernas;
- conflitos entre Academias;
- consequências regionais pós-campanha.

Este crescimento deve ser cuidadosamente controlado para não revelar demais cedo demais.

## 4.4 Crescimento Operacional

Aumenta capacidade do produto de se manter vivo.

Exemplos:

- temporadas;
- eventos;
- Frentes de Colapso rotativas;
- operações semanais;
- desafios especiais;
- recompensas cosméticas;
- mutadores de combate;
- rankings não-predatórios.

Este crescimento depende de telemetria e capacidade real de produção.

## 4.5 Crescimento Social

Adiciona interação entre jogadores.

Exemplos:

- Células de Campo;
- compartilhamento de builds;
- rankings assíncronos;
- eventos coletivos;
- simulações comparativas;
- coop assíncrono.

Este crescimento é posterior e não deve ser obrigatório para jogar.

---

# 5. Ordem Recomendada de Expansão

A ordem recomendada de crescimento é:

1. estabilidade do jogo base;
2. melhorias de UX, onboarding e legibilidade;
3. expansão de operações repetíveis;
4. expansão de loot e buildcraft;
5. primeira temporada leve;
6. primeira região adicional;
7. primeira Frente de Colapso avançada;
8. campanha complementar;
9. sistemas sociais assíncronos;
10. expansão maior paga ou grande atualização de conteúdo.

Não se deve criar grande expansão antes de provar:

- core loop;
- combate;
- transmutação;
- Codex;
- economia;
- retenção saudável;
- capacidade de produzir conteúdo recorrente sem colapsar o escopo.

# 6. Expansões de Região

Regiões são a forma mais natural de expansão para AZOTH.

Uma nova região pode adicionar:

- bioma visual;
- condições de campo;
- tipo de Breach dominante;
- estrutura política local;
- relações com Academias;
- materiais exclusivos;
- ruínas próprias;
- eventos repetíveis;
- criaturas ou fenômenos;
- entradas de Codex;
- operações de campanha;
- endgame regional.

## 6.1 Requisitos de Uma Nova Região

Uma região só deve entrar no produto se trouxer pelo menos três diferenciais reais:

1. nova leitura de Breach;
2. novo problema operacional;
3. nova consequência regional;
4. nova categoria de recurso;
5. novo dilema narrativo;
6. novo padrão visual forte;
7. nova interação tática;
8. nova hipótese de Codex.

Se a região for apenas “mais missões em outro cenário”, ela não deve ser priorizada.

## 6.2 Estrutura Mínima de Região Expandida

Uma região pós-lançamento deve conter, no mínimo:

- 1 hub ou ponto de contato regional;
- 1 linha narrativa curta;
- 3 a 5 operações de campanha ou investigação;
- 3 tipos de operação repetível;
- 1 ruína ou estrutura de campo relevante;
- 1 cadeia de Codex;
- 1 conjunto de materiais próprios;
- 1 ameaça ou fenômeno dominante;
- 1 recompensa de buildcraft associada;
- 1 consequência regional persistente.

## 6.3 Região Não Deve Ser

Uma região não deve ser:

- apenas mapa novo;
- apenas pacote de inimigos;
- apenas farm de recurso;
- apenas cenário para temporada;
- apenas conteúdo visual sem consequência.

Em AZOTH, lugar precisa ter função sistêmica e narrativa.

# 7. Expansão de Campanha

A campanha principal deve ser estruturada para permitir crescimento por arcos.

Tipos de expansão narrativa aceitáveis:

- arco regional;
- arco de Academia;
- arco de personagem;
- arco de ruína;
- arco de investigação;
- arco pós-revelação;
- arco de consequência.

## 7.1 Campanha Base vs Expansão

A campanha base deve entregar uma experiência completa, mas não deve esgotar o universo.

Ela deve:

- apresentar o jogador ao papel de agente alquímico;
- estabelecer Breach como fenômeno jogável;
- introduzir conflitos institucionais;
- revelar uma camada importante da verdade;
- abrir perguntas maiores;
- justificar endgame e operações contínuas.

Expansões podem:

- responder perguntas secundárias;
- aprofundar Academias;
- revelar contradições documentais;
- explorar ruínas específicas;
- mostrar consequências pós-campanha;
- introduzir novos perigos sem reescrever o núcleo.

## 7.2 Expansão Não Deve Corrigir Campanha Incompleta

A campanha base não pode depender de expansão para parecer completa.

Expansões devem ampliar, não remendar.

Se uma revelação é necessária para dar sentido ao jogo base, ela pertence ao jogo base.

Se uma revelação aprofunda entendimento, ela pode pertencer a expansão.

---

# 8. Expansão de Endgame

O endgame é o lugar mais importante para crescimento de longo prazo.

Ele deve sustentar:

- grind infinito saudável;
- chase loot;
- builds avançadas;
- teste de domínio tático;
- temporadas;
- Frentes de Colapso;
- mutadores;
- recompensas aspiracionais;
- Codex avançado;
- replayability.

## 8.1 Função do Endgame

O endgame deve responder:

> “O que eu faço depois que entendi o jogo e terminei o arco principal?”

A resposta não deve ser apenas:

> “Repita missões para números maiores.”

A resposta correta é:

> “Entre em zonas mais instáveis, teste builds mais autorais, persiga padrões raros, aprenda novas leituras de Breach e refine sua relação com a alquimia.”

## 8.2 Frentes de Colapso como Base de Endgame

Frentes de Colapso devem ser o principal sistema expansível de endgame.

Cada Frente pode combinar:

- sequência de combates;
- eventos de campo;
- risco crescente;
- decisão de extrair ou continuar;
- modificadores de Breach;
- recompensas progressivas;
- chase loot próprio;
- entradas de Codex;
- ranking opcional;
- variante sazonal.

## 8.3 Tipos de Frentes de Colapso

Exemplos de tipos:

### Frente de Ruído

Foco em falhas, instabilidade e previsão incompleta.

### Frente de Umbra

Foco em zonas escurecidas pelo Véu, risco de interpretação e contenção difícil.

### Frente de Ruína

Foco em exploração, quebra de padrão e artefatos antigos.

### Frente Orgânica

Foco em mutações, organismos deslocados e biomodificação.

### Frente de Auditoria

Foco em operações contra consequências de uso indevido da alquimia institucional.

### Frente de Convergência

Foco em múltiplos padrões de Breach simultâneos.

## 8.4 Crescimento de Endgame por Camadas

O endgame deve crescer em camadas:

1. operação repetível simples;
2. Frente de Colapso básica;
3. mutadores semanais;
4. chase loot por Frente;
5. Frentes com objetivos múltiplos;
6. Frentes sazonais;
7. Frentes narrativas;
8. Frentes de alta dificuldade;
9. Frentes com leaderboard opcional;
10. Frentes cooperativas assíncronas futuras.

---

# 9. Chase Loot em Expansões

Cada expansão relevante deve ter algum tipo de recompensa aspiracional.

Mas chase loot precisa ser controlado.

## 9.1 Categorias Recomendadas

- Protocolos Perdidos;
- Runas Irregulares;
- Matrizes de Transmutação raras;
- Catalisadores impossíveis;
- Instrumentos alquímicos únicos;
- padrões visuais de ET;
- cosméticos de agente;
- registros de Codex raros;
- títulos operacionais;
- emblemas de temporada;
- variantes de operação.

## 9.2 Chase Loot Não Deve

- ser obrigatório para vencer campanha;
- comprar poder estrutural;
- invalidar builds existentes;
- gerar uma única build dominante;
- exigir grind abusivo;
- depender de monetização;
- desaparecer para sempre se for mecanicamente relevante.

## 9.3 Chase Loot Deve

- gerar desejo;
- ter identidade visual e mecânica;
- contar algo sobre o mundo;
- abrir possibilidades;
- recompensar domínio;
- sobreviver a temporadas quando for importante;
- respeitar pity, fragmentos ou caminhos alternativos quando necessário.

# 10. Expansão de Buildcraft

Buildcraft é um dos principais motores de crescimento do produto.

Novas expansões podem adicionar:

- Domínios alquímicos especializados;
- Operações novas;
- modificadores de ET;
- geometrias;
- runas;
- catalisadores;
- instrumentos;
- especializações de agente;
- sinergias de equipe;
- efeitos condicionais de Codex;
- protocolos avançados.

## 10.1 Regra de Expansão de Build

Toda nova opção de build deve responder:

- que estilo ela favorece?
- que trade-off ela impõe?
- que problema ela resolve?
- que risco ela cria?
- que conteúdo ela torna mais interessante?
- como ela evita invalidar opções anteriores?

Se uma nova opção só aumenta número, ela é fraca como expansão.

## 10.2 Tipos de Build a Estimular

Expansões devem permitir arquétipos como:

- contenção segura;
- contenção agressiva;
- decomposição de precisão;
- controle de terreno;
- manipulação de instabilidade;
- extração de recursos;
- investigação avançada;
- suporte de equipe;
- alto risco / alta recompensa;
- builds de Codex;
- builds anti-Breach específico;
- builds sazonais.

## 10.3 Evitar Power Creep

Para evitar power creep:

- priorizar efeitos condicionais;
- usar trade-offs claros;
- limitar efeitos universais;
- manter raridade como identidade, não como licença para quebrar o jogo;
- criar counters naturais;
- revisar uso real por telemetria;
- evitar buffs constantes por pressão de comunidade.

# 11. Expansões de Sistemas

Nem todo sistema novo deve virar prioridade.

Sistemas só devem ser expandidos se melhorarem o core.

## 11.1 Sistemas Prioritários para Crescimento

- operações repetíveis;
- Frentes de Colapso;
- Engenharia de ETs;
- Codex jogável;
- pesquisa aplicada;
- loot/crafting;
- mutadores;
- estados regionais;
- eventos investigativos;
- compartilhamento de builds.

## 11.2 Sistemas Secundários

- decoração de hub;
- coleções cosméticas;
- ranking avançado;
- recursos sociais;
- desafios diários elaborados;
- simulações acadêmicas complexas;
- coop assíncrono.

## 11.3 Sistemas Perigosos

- PvP competitivo direto;
- mundo aberto;
- gacha de personagem;
- economia entre jogadores com poder;
- crafting irrestrito sem controle;
- procedural infinito sem curadoria;
- sistemas que exigem backend complexo cedo demais;
- eventos que exigem produção semanal intensiva.

# 12. Expansões de Codex

O Codex Veritatis deve crescer junto com o produto.

Novas expansões devem adicionar:

- novas entradas;
- novas hipóteses;
- contradições;
- arquivos restritos;
- camadas de interpretação;
- desbloqueios sistêmicos;
- vantagens investigativas;
- recontextualizações.

## 12.1 Tipos de Expansão de Codex

### Entrada Regional

Explica fenômenos específicos de uma região.

### Entrada de Breach

Classifica novos padrões de instabilidade.

### Entrada de Academia

Mostra interpretação institucional e conflito político-científico.

### Entrada de Ruína

Registra vestígios pré-modernos ou estruturas esquecidas.

### Entrada de Artefato

Conecta loot raro a história sistêmica.

### Entrada Sazonal

Registra fenômenos temporários que podem virar permanentes.

## 12.2 Codex Como Recompensa de Expansão

Uma expansão deve recompensar o jogador com entendimento.

Não basta dar item.

O jogador deve sentir:

> “Agora eu entendo uma parte do mundo que antes parecia apenas mecânica.”

---

# 13. Expansões de Personagens

Novos personagens devem ser introduzidos com cuidado.

Um personagem novo precisa justificar:

- função narrativa;
- função mecânica;
- relação com Academia/região;
- visão sobre alquimia;
- conflito ou dilema próprio;
- impacto no hub ou em operações;
- razão para existir além de coleção.

## 13.1 Personagem Não Deve Ser Conteúdo Gacha

Personagens de AZOTH não devem existir como pacote de monetização de poder.

Eles podem ser:

- companheiros desbloqueáveis por campanha;
- especialistas regionais;
- contatos de Academia;
- operadores de suporte;
- NPCs de hub;
- personagens de expansão narrativa;
- skins ou variantes cosméticas sem poder.

## 13.2 Expansão de Equipe

A equipe jogável deve crescer devagar.

Adicionar muitos personagens aumenta custo de:

- escrita;
- animação;
- balanceamento;
- UI;
- sinergia;
- testes;
- onboarding.

Para produção solo, poucos personagens fortes valem mais do que muitos personagens rasos.

---

# 14. Expansões de Academias

Academias são excelente vetor de expansão.

Elas podem adicionar:

- linhas de pesquisa;
- reputação;
- protocolos;
- dilemas institucionais;
- missões específicas;
- interpretações conflitantes;
- acesso a materiais;
- restrições éticas;
- especializações de build.

## 14.1 Academia Como Produto

Uma Academia nova ou expandida deve trazer:

- identidade visual;
- vocabulário próprio;
- visão de mundo;
- impacto em gameplay;
- linha de missões;
- recompensa sistêmica;
- conflito com outras instituições.

## 14.2 Risco

Academias não podem virar “facções genéricas de RPG”.

Elas devem representar formas diferentes de interpretar e institucionalizar a alquimia.

---

# 15. Expansões Sazonais que Viram Permanentes

Algumas temporadas podem servir como teste de futuras expansões.

Uma temporada pode virar conteúdo permanente se:

- teve boa retenção;
- teve feedback qualitativo forte;
- criou tipo de operação interessante;
- adicionou mutador saudável;
- não gerou economia tóxica;
- não quebrou balanceamento;
- tem função canônica duradoura;
- não exige manutenção excessiva.

## 15.1 Exemplo de Conversão

Uma temporada chamada `Maré de Ruído` pode introduzir mutadores de Ruído.

Se esses mutadores forem bons, eles podem virar:

- Frente de Colapso permanente;
- cadeia de Codex;
- categoria de operação repetível;
- conjunto de recompensas;
- expansão regional.

## 15.2 Temporada Não Deve Forçar Permanência

Nem todo evento bom precisa virar sistema permanente.

Às vezes o melhor uso de um evento é ser memória limitada, recompensa cosmética e aprendizado operacional.

---

# 16. Expansões Pagas

Expansões pagas são permitidas, desde que respeitem `05.6`.

Podem incluir:

- nova campanha regional;
- nova região;
- nova Academia;
- novas operações;
- novos personagens narrativos;
- novos inimigos/fenômenos;
- novos itens e ETs;
- novos cosméticos;
- novas Frentes de Colapso;
- novas camadas de Codex.

## 16.1 Expansão Paga Não Deve

- vender poder obrigatório;
- bloquear correções de qualidade;
- bloquear sistemas básicos;
- impedir o jogador base de continuar usando o endgame principal;
- fragmentar matchmaking essencial;
- prometer revelação central fora do jogo base;
- parecer conteúdo removido do lançamento.

## 16.2 Melhor Formato de Expansão Paga

O melhor formato é:

> **região + campanha curta + operações repetíveis + Frente de Colapso + Codex + recompensas de buildcraft + cosméticos temáticos.**

Esse pacote entrega valor claro sem depender de monetização predatória.

---

# 17. Atualizações Gratuitas

Atualizações gratuitas devem sustentar confiança.

Podem incluir:

- melhorias de UX;
- balanceamento;
- correções;
- novas operações repetíveis pequenas;
- eventos limitados;
- recompensas cosméticas;
- entradas menores de Codex;
- novos mutadores;
- melhorias de performance;
- ajustes de onboarding.

## 17.1 Regra

Correção, qualidade e integridade sistêmica não devem ser tratadas como conteúdo pago.

# 18. Crescimento de Comunidade

A comunidade deve crescer em torno de:

- builds;
- descobertas;
- teorias;
- Codex;
- estratégias de operação;
- chase loot;
- eventos;
- desafios;
- arte e identidade do mundo.

## 18.1 Conteúdos Compartilháveis

O produto deve favorecer compartilhamento de:

- builds de ET;
- resultados de Frentes de Colapso;
- screenshots de círculos de transmutação;
- descobertas de Codex;
- layouts de equipe;
- relatos de operação;
- loot raro;
- desafios sazonais;
- decisões narrativas.

## 18.2 Comunidade Não Deve Ser Obrigatória

O jogador solo precisa continuar tendo experiência completa.

Comunidade é amplificação, não pré-requisito.

---

# 19. Crescimento de Conteúdo Produzido com IA

Como o projeto será produzido por um desenvolvedor solo com apoio de IA, expansão precisa respeitar um pipeline modular.

IA pode ajudar em:

- variações de eventos;
- rascunhos de missões;
- descrições de Codex;
- conceitos visuais;
- ícones;
- nomes de itens;
- protótipos de UI;
- balanceamento preliminar;
- análise de telemetria;
- geração de documentação;
- teste de cenários.

IA não deve substituir:

- direção criativa;
- revisão canônica;
- tuning final;
- curadoria narrativa;
- validação de diversão;
- decisão de escopo.

## 19.1 Regra de Curadoria

Todo conteúdo gerado com apoio de IA precisa passar por filtro de:

1. cânone;
2. gameplay;
3. legibilidade;
4. viabilidade;
5. redundância;
6. valor real para o jogador.

---

# 20. Critérios para Aprovar Uma Expansão

Uma expansão só deve ser aprovada se responder positivamente à maioria das perguntas abaixo:

- aprofunda o core fantasy?
- cria novas decisões táticas?
- adiciona novos usos de transmutação?
- expande Codex de forma relevante?
- respeita narrativa e revelação?
- aumenta replayability?
- oferece recompensa desejável?
- cabe na capacidade de produção?
- não depende de monetização predatória?
- não quebra o balanceamento existente?
- tem identidade visual própria?
- melhora retenção sem criar obrigação abusiva?
- pode ser testada em pequena escala?
- pode ser cortada sem destruir o produto?

Se a resposta principal for “precisamos de mais conteúdo”, a expansão ainda não está bem justificada.

---

# 21. Critérios para Não Expandir

Não expandir quando:

- core loop ainda não está validado;
- jogadores não entendem transmutação;
- combate está cansativo;
- Codex está sendo ignorado;
- economia está quebrada;
- onboarding está fraco;
- produção está atrasada;
- telemetria indica abandono cedo;
- nova expansão depende de retrabalho estrutural;
- comunidade pede algo que contradiz a identidade;
- monetização está pressionando design.

Nesses casos, melhorar base vale mais do que adicionar conteúdo.

---

# 22. Roadmap de Crescimento Recomendado

## 22.1 Após MVP Interno

Foco:

- corrigir core loop;
- melhorar combate;
- validar ETs;
- melhorar UX;
- validar Codex mínimo;
- testar grind repetível.

Não expandir ainda.

## 22.2 Após Vertical Slice

Foco:

- ampliar região inicial;
- adicionar variações de operação;
- testar loot raro;
- introduzir uma Frente de Colapso simples;
- validar primeira cadeia de Codex.

## 22.3 Após Alfa

Foco:

- campanha base curta;
- progressão consistente;
- economia inicial;
- endgame mínimo;
- telemetria;
- conteúdo repetível saudável.

## 22.4 Após Beta

Foco:

- balanceamento;
- retenção;
- comunicação;
- soft launch;
- ajustes de monetização;
- primeira temporada controlada.

## 22.5 Após Lançamento Inicial

Foco:

- estabilidade;
- feedback;
- correções;
- pequenos eventos;
- expansão de operações;
- preparar primeira grande expansão.

## 22.6 Após Primeira Temporada Bem-sucedida

Foco:

- transformar melhores mutadores em conteúdo permanente;
- adicionar chase loot adicional;
- expandir Frentes de Colapso;
- planejar nova região ou arco narrativo.

# 23. Formato Recomendado de Grande Expansão

Uma grande expansão deve ter estrutura padronizada:

1. título temático;
2. premissa canônica;
3. nova região ou zona;
4. novo fenômeno de Breach;
5. linha narrativa;
6. operações de campanha;
7. operações repetíveis;
8. Frente de Colapso;
9. Codex expandido;
10. loot e crafting;
11. novas opções de build;
12. recompensas cosméticas;
13. mutadores;
14. telemetria dedicada;
15. critérios de sucesso.

## 23.1 Exemplo de Estrutura

### Expansão: Ecos de Umbra

Conteúdo possível:

- região marcada por distorções do Véu;
- nova linha de investigação;
- operações de contenção de baixa visibilidade;
- runas ligadas a reversibilidade parcial;
- Frente de Colapso de Umbra;
- Codex sobre interpretações conflitantes;
- chase loot: Instrumento de leitura de Véu;
- cosméticos de círculo escurecido;
- mutadores de percepção.

Este exemplo é direcional, não conteúdo canônico fechado.

---

# 24. Formato Recomendado de Expansão Menor

Uma expansão menor pode conter:

- 1 cadeia curta de operações;
- 1 novo mutador;
- 1 conjunto pequeno de recompensas;
- 1 nova entrada de Codex em camadas;
- 1 variação de Frente ou Incidente;
- 1 cosmético temático;
- ajustes de balanceamento.

Expansões menores são úteis para produção solo porque permitem aprendizado sem grande risco.

---

# 25. Relação com Monetização

Expansão e monetização devem seguir `05.6`.

Permitido:

- vender campanha/região adicional;
- vender cosméticos;
- vender passe de temporada ético;
- vender pacote fundador;
- vender DLC pequeno de conteúdo;
- vender conveniência leve sem poder estrutural.

Proibido:

- vender chance de chase loot mecânico;
- vender recurso competitivo;
- vender salto de progressão que destrua aprendizado;
- vender revelação central;
- vender builds dominantes;
- vender ET superior obrigatória;
- vender energia para jogar o core.

## 25.1 Regra

Jogador pode pagar para ter mais AZOTH.

Jogador não deve pagar para entender menos, jogar menos ou vencer sem domínio.

---

# 26. Relação com Telemetria

A decisão de expandir deve usar telemetria, mas não ser escrava dela.

Métricas úteis:

- retenção por fase;
- conclusão de campanha;
- repetição de operações;
- uso de ETs;
- diversidade de builds;
- abandono por operação;
- tempo em Codex;
- taxa de crafting;
- uso de Frentes de Colapso;
- taxa de extração vs falha;
- busca por chase loot;
- compra de conteúdo;
- retorno em temporada;
- feedback qualitativo.

## 26.1 Sinais Positivos

- jogadores repetem operações sem obrigação;
- builds variadas aparecem;
- Codex é consultado depois de missões;
- Frentes geram “só mais uma run”;
- chase loot gera desejo sem revolta;
- jogadores discutem teorias;
- jogadores compartilham builds;
- monetização não domina feedback.

## 26.2 Sinais de Alerta

- todos usam a mesma build;
- jogadores ignoram Codex;
- endgame vira farm automático;
- recompensas parecem insuficientes;
- grind parece obrigatório;
- temporada gera FOMO agressivo;
- expansão divide comunidade;
- nova região tem baixa conclusão;
- conteúdo novo é consumido e abandonado rápido.

---

# 27. Crescimento Sem Quebrar Jogador Novo

Quanto mais o jogo cresce, maior o risco de intimidar novos jogadores.

Para evitar isso:

- campanha base deve continuar clara;
- primeiras ETs devem permanecer relevantes;
- onboarding deve ser atualizado;
- sistemas avançados devem ser desbloqueados gradualmente;
- expansão não deve jogar dez sistemas novos de uma vez;
- Codex deve organizar conhecimento por camadas;
- builds recomendadas devem existir sem remover liberdade;
- conteúdo antigo deve ser revisitado e melhorado.

## 27.1 Reentrada de Jogador Retornando

Jogadores que voltam após meses devem receber:

- resumo de mudanças;
- indicação do que fazer agora;
- acesso a operações úteis;
- caminho para recuperar conteúdo essencial;
- explicação de temporadas encerradas;
- destaque de recompensas ainda disponíveis.

---

# 28. Crescimento e Complexidade

AZOTH pode ficar complexo. Isso é parte do apelo.

Mas complexidade precisa ser progressiva.

Complexidade boa:

- cria decisão;
- gera domínio;
- recompensa estudo;
- mantém clareza;
- permite experimentação.

Complexidade ruim:

- exige planilhas para jogar casualmente;
- esconde regra essencial;
- pune sem explicar;
- depende de wiki externa;
- confunde risco com aleatoriedade injusta.

---

# 29. Riscos de Crescimento

## 29.1 Power Creep

Risco de novos itens invalidarem antigos.

Mitigação:

- foco em poder horizontal;
- trade-offs;
- balanceamento por telemetria;
- limites de raridade;
- rotação de mutadores.

## 29.2 Escopo Incontrolável

Risco de prometer mais do que pode ser produzido.

Mitigação:

- expansão modular;
- vertical slice de expansão;
- protótipos antes de campanha grande;
- cortes planejados.

## 29.3 Lore Inflation

Risco de revelar demais ou adicionar mistérios artificiais.

Mitigação:

- revisão pela Constituição;
- amarração com Codex;
- revelação em camadas;
- não usar segredo como muleta.

## 29.4 Grind Abusivo

Risco de endgame virar obrigação.

Mitigação:

- recompensas alternativas;
- pity/fragmentos quando necessário;
- cosméticos como chase secundário;
- teto semanal suave sem punir ausência.

## 29.5 Produção Solo Sobrecarregada

Risco de live ops exigir cadência impossível.

Mitigação:

- temporadas menores;
- reutilização inteligente;
- mutadores sistêmicos;
- eventos com variação de regras, não só conteúdo novo;
- ferramentas internas.

---

# 30. Estratégia de Crescimento Recomendável Para Produção Solo

Para um desenvolvedor solo com apoio de IA, a ordem saudável é:

1. expandir sistemas já prontos antes de criar sistemas novos;
2. criar novas operações usando estruturas existentes;
3. criar mutadores em vez de modos inteiros;
4. criar recompensas de build antes de novos personagens;
5. criar Codex novo com impacto sistêmico leve;
6. criar região pequena antes de região grande;
7. testar temporadas como laboratório;
8. só criar expansão paga quando o pipeline estiver previsível.

## 30.1 Conteúdo Mais Barato e Valioso

- novas operações com objetivos diferentes;
- novos mutadores;
- novas entradas de Codex;
- novas runas;
- novos catalisadores;
- novos padrões visuais de ET;
- novas recompensas cosméticas;
- novas combinações de Breach;
- variações de Frentes de Colapso.

## 30.2 Conteúdo Mais Caro

- nova região com arte completa;
- novos personagens jogáveis;
- campanha longa;
- multiplayer síncrono;
- cinemáticas;
- sistemas inéditos;
- nova plataforma;
- nova linguagem/localização extensa.

# 34. Decisões Fechadas

Ficam definidas as seguintes decisões:

1. Expansão deve priorizar profundidade antes de volume.
2. Regiões são o principal vetor de expansão de conteúdo.
3. Frentes de Colapso são o principal vetor de expansão de endgame.
4. Buildcraft e Codex devem crescer junto com qualquer expansão relevante.
5. Temporadas podem servir como laboratório para sistemas permanentes.
6. Chase loot deve existir, mas não pode ser obrigatório nem monetizado como poder.
7. Expansões pagas são aceitáveis se entregarem conteúdo real e não venderem poder estrutural.
8. Crescimento social é posterior e deve permanecer solo-first.
9. Produção solo exige expansão modular, reutilização inteligente e gates de corte.
10. Nenhuma expansão deve corrigir falhas fundamentais do jogo base; primeiro corrige-se a base.

# 37. Princípio Final

AZOTH: BREACH deve crescer como um organismo alquímico controlado: cada nova camada precisa preservar estrutura, custo, consequência e identidade.

Crescer é fazer o jogador sentir que Aletheia continua profunda, instável, recompensadora e coerente — sem que o produto perca a forma que o tornou especial.
